# 仓库守则（REPO-RULES）

> 本文件是本仓库的最高规则文件，所有贡献者（包括 AI）必须遵守。

---

## 1. 文档管理

- **仓库级长期记忆、项目状态与全局知识，统一存放在 `docs/` 目录。**
- 禁止将仓库级长期记忆沉淀到仓库外部、AI 记忆系统的非 repo 范围，或其他任何非 `docs/` 目录。
- `docs/` 目录结构说明：

  | 路径 | 用途 |
  |---|---|
  | `docs/REPO-RULES.md` | 本守则，仓库最高规则 |
  | `docs/architecture.md` | 技术架构与模块设计 |
  | `docs/README.md` | 项目概述与功能说明 |
  | `docs/updates/` | 时效性更新记录（按日期命名） |

## 2. 更新记录

- **任意更新的时效性记录，统一写入 `docs/updates/<日期>-update.md`**（例如 `2026-06-01-update.md`）。
- 不要将更新记录写入 repo-memory、流程手册或其他文档。
- 每次更新记录必须包含：变更内容摘要、影响范围、关联文件列表。

## 3. 代码与文档同步

- 代码或规则发生变化后，必须同步更新相关 `docs/` 文档，避免文档与实现脱节。
- PR / 提交合并时，相关文档应在同一次提交中更新。

## 4. 技术规范

- 项目为**纯前端**实现，禁止引入任何后端服务依赖。
- 技术栈：React 18 + TypeScript + Vite 6 + Tailwind CSS v4 + Zustand + @dnd-kit + pdf-lib + pdfjs-dist。
- JavaScript / TypeScript 使用 ES Module 风格，不使用 CommonJS。
- 包管理器统一使用 **bun**（不使用 npm / yarn / pnpm）。
- 所有新增第三方依赖必须同步写入 `package.json`，不允许仅在代码中 `import` 而不声明依赖。

## 5. 安全规范

- 所有文件处理均在浏览器本地完成，不上传到任何服务器。
- 不在代码中硬编码任何 API Key、Token 或敏感信息。
- 文件读取使用 `FileReader` / `File API`，不依赖 Node.js 文件系统。

## 6. 呈交前审阅（AI 强制规则）

> **任何代码变更在呈交给用户之前，必须完成以下检查，缺一不可：**

1. **依赖完整性**：所有在代码中 `import` 的第三方包，必须已存在于 `package.json` 的 `dependencies` 或 `devDependencies` 中。
2. **构建可通过**：运行 `bun run build`（或 `bun run dev` 观察无 error）确认无编译错误后，方可呈交。
3. **Vite 动态 import**：对运行时才确定的动态导入（如 `import('pkg-name')`），必须加注释 `/* @vite-ignore */` 以避免 Vite 静态分析失败。
4. **类型安全**：不使用 `'string' as any` 绕过类型检查；应将 `as any` 施加在模块结果上，而非导入路径字符串上。

## 7. 浏览器验证（AI 强制规则）

> **每次修改功能或修复 Bug 之后，在向用户报告「已完成/已修复」之前，必须亲自用浏览器验证结果，确认功能符合预期。未经验证不得声称修复完成。**

验证流程：
1. 确认开发服务器正在运行（`bun run dev`）。
2. 使用浏览器工具打开 `http://localhost:5173/`，截图或检查页面状态。
3. 执行与本次改动相关的具体操作（上传文件、触发功能等），观察结果是否正确。
4. 如发现问题，继续修复并重新验证，直到通过后方可呈交。
